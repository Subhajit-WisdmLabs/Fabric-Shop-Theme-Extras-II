Print affordance term (OPEN — for Nakul)
  • The issue: in print/design, "print-ready" conventionally describes artwork/files prepared for output — not a base fabric that can receive print. So it's a mild misnomer for a stocked base. Worth a look before it's locked across the badge, the per-SKU metafield, PrintLab, and Custom.
  • Options: "Print base" — the actual textile-trade term for a fabric you print onto; two words, fits trade-not-tech voice, but reads as industry shorthand. · "Printable" — plainest, instantly understood by any maker, slightly utilitarian. · "Print to order" — describes the action, ties to POD. · "Print-ready" — current; familiar but the file-not-fabric ambiguity.
  • Recommendation: "Print base" for the pill (trade-accurate, on-voice), with the legend line carrying the plain-language explanation. If customer clarity outranks register, "Printable" is the safe pick. Either way the legend does the teaching, so the pill can stay terse.
  • Whatever's chosen replaces the prior "Print-ready (one word, hyphenated)" lock and propagates to: card pill, the "Print →" hover link, the per-SKU metafield label, PrintLab + Custom references, and this legend.

Filter taxonomy → schema map
  • Category → yard.tag_category (radio, REQUIRED). Enum: fabric / button / zip / notion. Drives template selection at PDP.
  • Fabric sub-type → yard.fabric_subtype (single, only when category=fabric). Enum: woven / knit / lining / deadstock_print. Rendered as sub-chips beneath the Fabrics radio.
  • Print compatible → yard.print_compatible (boolean). Drives the card-level "Print-ready" badge (signal only). v8: the price-row "Print →" link and the thumb hover-strip were removed — the print action now lives on the Yard PDP ("Print this fabric" CTA), reinforced by the mid-grid PrintLab prompt. NOT a filter facet — discovery via card badge only (Nick's call).
  • Fibre → yard.fibre[] (multi). Cotton / linen / silk / wool / hemp / bamboo / viscose. Available when category=fabric.
  • Weight → yard.weight_band (single) + yard.gsm numeric. Chips drive band; range drives precise GSM. Available when category=fabric.
  • Drape → yard.drape[] (multi, max 2). Soft & flowing / crisp / structured / stable. Available when category=fabric.
  • Width → yard.width_cm (single, from variant). Chips group nearest standards. Available where width is relevant (fabrics, bias binding, twill tape, ribbing).
  • Colour → yard.colour_palette[] (hex array). CIE Lab distance ≤ 25 against tap. Notions inherit from variant-colour.
  • Project → yard.tag_project[] (multi, max 4). Shirting / dressmaking / tailoring / bag-making / quilting / outerwear / children's / loungewear / accessory / soft-furnishing / t-shirts (added v4 — couples to the knit additions). The bundle key — same taxonomy as the discovery rail.
  • Care → yard.tag_care[] (multi). Machine wash / hand wash / dry clean.
  • Sold as → derived from yard.unit_of_sale. Metre / piece / pack.
  • Availability → derived. published_at > now-30d = new; restocked_at > now-30d = restocked; stock_level < threshold = low; fabric_subtype = deadstock_print = one of a kind (badge inherent, badge persistence dependent on stock_level).
Filter UX rules
  • Category: radio (one only) — exclusive. Other filters reactively adapt their counts and available options to the chosen category.
  • Sub-chips under Fabrics (Wovens / Knits / Linings / Deadstock prints): single-select, optional. Default is "all fabrics". Selecting a sub-chip further narrows the count and adjusts the fibre/weight/drape facets to those relevant to the sub-type (e.g. selecting Linings collapses Weight; selecting Deadstock prints surfaces "Origin" pivot instead of Fibre).
  • When Category = Buttons / Zips / Notions: Fibre / Weight / Drape / GSM filters collapse silently (irrelevant). Width filter retained where applicable (e.g. bias binding widths, twill tape widths, ribbing widths).
  • Fibre / Drape / Project / Care: checkbox arrays, OR within group, AND across groups.
  • Weight chip and GSM range are linked but mutually exclusive — picking a chip locks the range; sliding the range clears the chip.
  • Colour: tap swatch → backend uses CIE Lab distance ≤ 25 against yard.colour_palette. Multi-select OK.
  • All filter state mirrored into URL query string — shareable, bookmarkable.
  • Helper line at top of sidebar is permanent (not collapsible): "In stock now. Everything in The Yard ships within 3 business days.*" with footnote "*Orders combined with a PrintLab print follow PrintLab's lead times." (v8: dropped "from our Melbourne studio" — we don't need the specificity, and combined PrintLab orders follow PrintLab lead times.)

Filter set · v6 launch trim
  • Principle: a 62-fabric launch range can't fill a long sidebar without facet values that read as broken ("Hemp (1)") and an implied catalogue depth that isn't there. Ship fewer, fuller facets; restore the rest as the range grows. 3-SKU floor: don't render a facet value with <3 SKUs behind it; collapse a whole group if it has <2 populated values.
  • Launch set (3): Category (+ fabric sub-chips: Wovens / Knits / Linings / Deadstock prints) · Fibre (Cotton / Linen / Silk / Wool) · Weight (bands). Plus persistent Search + Sort.
  • Removed from the surface in v6: Width (a PDP spec at this range) · Colour (weak axis while bases are mostly naturals) · Sold-as (duplicates Category — buttons=piece, fabric=metre, pack=pack) · Availability (New / Restocked / Low-stock — merchandising; "Newest first" and "Recently restocked" already live in Sort).
  • Already deferred (v5): Drape · Care (lives on PDP) · GSM precision slider (bands suffice) · Project facet · rare fibres hemp/bamboo/viscose (under floor).
  • Every removed/deferred facet stays in the data model (yard.width, yard.colour[], sold_as, yard.availability, yard.drape[], yard.tag_care[], yard.gsm, yard.tag_project[]) — only the UI is withheld. Re-enabling each is a section-settings toggle, not a schema change or re-tag. Nat/Subhajit flip them on per-facet as inventory crosses the floor.
  • Facet counts computed server-side; the floor is enforced in the facet-builder, so values appear/disappear automatically as inventory crosses the threshold. No manual maintenance.
